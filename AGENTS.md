# Agent context (for LLMs + automation)

This repository is a pnpm workspaces monorepo for an MVP aquaponics monitoring system deployed across roughly 30 sites. It ingests device telemetry, stores normalized sensor measurements, detects explainable anomalies, shows site health in an admin dashboard, and sends critical alert emails.

If you are an automated agent, read these docs/files in order:

1. `README.md` - human setup, local development, ingestion example, Railway deployment.
2. `AGENTS.md` - this file; current repo status, implementation map, constraints, and maintenance expectations.
3. `packages/db/src/types.ts` - canonical database table/type map.
4. `packages/db/src/migrations/` - actual schema shape and indexes.
5. `apps/api/src/graphql/schema.graphql` - GraphQL contract used by the frontend.
6. `apps/api/src/server.ts` - API bootstrap, route mounting, scheduler startup.
7. `apps/web/app/(dashboard)/` - dashboard page structure.

## High-level goals

- Keep the backend simple: one Node/TypeScript service, no microservices, no message queues.
- Use GraphQL for frontend-facing data: reads, `updateMe`, and **admin-only** user/site CRUD (`adminUsers`, `adminSites`, `createAdminUser`, `updateAdminUser`, `resetAdminUserPassword`, `createAdminSite`, `updateAdminSite`). Do not move device ingestion into GraphQL.
- Use REST only for device ingestion at `POST /ingest`.
- Store sensor readings as normalized time-series rows, one row per sensor reading.
- Enforce RBAC in backend resolvers; frontend visibility is only a convenience.
- Deploy the whole system on Railway: Railway PostgreSQL, one API service, one web service.
- Keep the database plain PostgreSQL initially while preserving a low-friction TimescaleDB upgrade path.
- Prefer clear, maintainable code over premature abstraction.

## Environment variables

Root `.env.example` documents expected variables. For **local development**, pnpm runs `migrate`/`seed`/`api`/`web` from each package directory, so each process loads env from its own cwd:

- `packages/db/.env` — `pnpm migrate`, `pnpm seed`
- `apps/api/.env` — API and `pnpm dev` for `@app/api`
- `apps/web/.env.local` — Next.js (`@app/web`)

Keep the same values in all three (or symlink). Committed templates: `packages/db/.env.example`, `apps/api/.env.example`, `apps/web/.env.example`.

Shared:

- `DATABASE_URL` - Railway PostgreSQL or local Postgres connection string.
- `PG_POOL_MAX` - optional; max connections in the shared `pg` pool (`packages/db` `createDb`). Default `10`. Use a lower value on the **API** Railway service (e.g. `3`) if you want slightly lower client memory; the web app can keep the default or set its own.

Web/Auth:

- `AUTH_SECRET` - required for Auth.js JWT signing; must be identical on web and API.
- `AUTH_URL` - deployed web URL on Railway, or `http://localhost:3333` locally.
- `NEXTAUTH_URL` - deployed web URL on Railway, or `http://localhost:3333` locally.
- `NEXT_PUBLIC_API_URL` - browser-visible API base URL, e.g. Railway API URL.

API:

- `AUTH_SECRET` - same as web; used with `getToken` from `next-auth/jwt` to resolve the user from the session cookie on GraphQL requests.
- `WEB_ORIGIN` - web URL allowed by API CORS in **production**. In **development** (`NODE_ENV !== "production"`), CORS reflects the request `Origin` so local hostname/port mismatches are less brittle; production remains a single explicit origin.
- `PORT` - Railway provides this; local fallback is `4000`.
- `RESEND_API_KEY` - required for real email delivery.
- `ALERT_FROM_EMAIL` - sender email for alert notifications.
- `COOLDOWN_MINUTES` - critical alert notification cooldown; default is `45`.

## Railway deployment

Create one Railway project with three services:

- PostgreSQL service providing `DATABASE_URL`.
- API service using repository root as the Railway root directory.
- Web service using repository root as the Railway root directory.

Do not set Railway root directories to `apps/api` or `apps/web`; both services need workspace access to `packages/db`.

API service:

- Build command: `pnpm build:api`
- Start command: `pnpm start:api`
- Required env: `DATABASE_URL`, `AUTH_SECRET`, `WEB_ORIGIN`, `AUTH_URL`, `RESEND_API_KEY`, `ALERT_FROM_EMAIL`
- Optional env: `COOLDOWN_MINUTES`

Web service:

- Build command: `pnpm build:web`
- Start command: `pnpm start:web`
- Required env: `DATABASE_URL`, `AUTH_SECRET`, `AUTH_URL`, `NEXTAUTH_URL`, `NEXT_PUBLIC_API_URL`
- Optional env: `NEXT_PUBLIC_GOOGLE_MAPS_EMBED_API_KEY` (Maps Embed API for site detail iframe; enable **Maps JavaScript API** on the same key for the admin site form’s draggable map picker)

Deployment order:

1. Create Railway PostgreSQL.
2. Deploy API.
3. Run `pnpm migrate:deploy`.
4. Optionally run `pnpm seed`.
5. Deploy web.
6. Set `WEB_ORIGIN` on API to the deployed web URL.
7. Set `NEXT_PUBLIC_API_URL` on web to the deployed API URL.

## Database schema overview

Implemented migrations:

- `0001_auth.ts`
  - Auth.js-compatible tables: `users`, `accounts`, `sessions`, `verification_tokens`.
  - Adds `users.role` with values `admin`, `site_manager`, `site_viewer`.
  - Adds `users.password_hash` for credentials login.

- `0002_core.ts`
  - `sites`
  - `devices`
  - `user_sites`
  - `sensor_thresholds` — optional per-site overrides for critical min/max (`sensor` matches `sensor_catalog.key` after migration `0009`)

- `0003_measurements.ts`
  - `measurements`
  - one row per sensor reading
  - composite primary key `(taken_at, id)`
  - indexes for `(site_id, sensor, taken_at desc)` and `(device_id, taken_at desc)`
  - schema is intentionally Timescale-ready but does not require TimescaleDB

- `0004_alerts.ts`
  - `alerts`
  - partial unique index for active `(site_id, type)` alert dedupe
  - partial active-alert index for health/status queries

- `0005_site_geo_sensors.ts`
  - `sites.latitude`, `sites.longitude` (optional GPS for map)
  - (historical) `sites.sensor_visibility` jsonb — **removed** in `0006_sensor_catalog.ts`

- `0006_sensor_catalog.ts`
  - `sensor_catalog` — canonical sensor keys, display names, units, physical min/max, sort order (seeded for the four MVP sensors)
  - `site_sensor_catalog` — one row per site per catalog sensor (`enabled`); backfills from legacy `sensor_visibility` then drops that column

- `0007_measurements_sensor_catalog_keys.ts`
  - drops the `measurements.sensor` check constraint so new catalog keys can be ingested without a migration

- `0008_devices_admin.ts`
 - adds optional `devices.name` (friendly label), `devices.board` (e.g. `wemos_d1_mini`), and `devices.pin_map` (jsonb of last sensor→GPIO mapping recorded at install time)
 - `device_id` and `api_key_hash` semantics on `/ingest` are unchanged; plaintext keys are still never stored

- `0009_sensor_thresholds_catalog_keys.ts`
  - drops the legacy check on `sensor_thresholds.sensor` so any catalog key can have per-site threshold overrides; effective bounds = override column if set, else `sensor_catalog.physical_min` / `physical_max`

## TimescaleDB upgrade path

TimescaleDB is not enabled for the MVP. The `measurements` table is shaped so a future migration can add TimescaleDB with minimal disruption:

```sql
CREATE EXTENSION IF NOT EXISTS timescaledb;
SELECT create_hypertable('measurements', 'taken_at', migrate_data => true);
```

Do not add unique constraints to `measurements` that exclude `taken_at`; that would break hypertable conversion.

## API entrypoints and flow

- `apps/api/src/server.ts`
  - creates the shared Kysely DB client
  - creates the email notifier
  - configures Fastify and CORS
  - mounts `/graphql`
  - mounts `/ingest`
  - starts scheduler

- `apps/api/src/auth/session.ts`
  - reads the Auth.js session cookie using `getToken` from `next-auth/jwt` and `AUTH_SECRET` (must match web)

- `apps/api/src/ingest/route.ts`
  - validates `x-api-key`
  - validates payload with zod (`readings` is a string-keyed record; keys must exist in `sensor_catalog`; unknown keys return 400)
  - stores normalized measurements
  - updates `devices.last_seen_at`
  - resolves active `device_offline` alerts
  - runs anomaly detection and alert upsert for sensors enabled in `site_sensor_catalog`; range checks use `sensor_thresholds` + catalog defaults (`apps/api/src/anomaly/thresholds.ts`); spike/flatline heuristics remain on the four MVP sensor keys only (`apps/api/src/anomaly/detect.ts`)

- `apps/api/src/site/sensor-reporting.ts`
  - maps alert types to sensor keys, `shouldSurfaceAlert` for dashboard filtering, coordinate validation; MVP sensor list for threshold/anomaly wiring

- `apps/api/src/graphql/schema.graphql`
  - source of truth for frontend GraphQL queries

- `apps/api/src/graphql/resolvers/`
  - `sites.ts`
  - `measurements.ts`
  - `alerts.ts`
  - `user.ts` (`getMe`, `updateMe`)
  - `admin.ts` (admin-only users/sites listing and mutations; uses `requireAdmin`)

- `apps/api/src/anomaly/`
  - threshold defaults and simple explainable anomaly rules

- `apps/api/src/alerts/`
  - alert upsert/dedupe
  - Resend email notifier
  - recommended action text

## GraphQL contract

Frontend-supported queries:

- `getSites`
- `getSite(id)` — includes optional `latitude` / `longitude`, `sensorReporting { key, displayName, unit, physicalMin, physicalMax, rangeMin, rangeMax, thresholdMinOverride, thresholdMaxOverride, enabled }` (effective `range*` = site overrides or catalog defaults; disabled sensors omitted from measurements/alerts in API responses and do not affect derived `status` / `lastUpdate`)
- `getMeasurements(siteId, range)`
- `getAlerts(siteId, type, status)`
- `getMe` (authenticated user profile)
- `sensorCatalog` — **`admin` role only**; global definitions (`id`, `key`, `displayName`, `unit`, `physicalMin` / `physicalMax`, `sortOrder`). The web app’s `AdminSites` operation also selects `sensorCatalog` in the same request for site form checkboxes.
- `adminUsers`, `adminSites` — **`admin` role only**; list users with `assignedSiteIds`, and all sites.
- `adminDevices(siteId)`, `adminDevice(id)` — **`admin` role only**; list/lookup devices for the install wizard and device manager (`siteId`, `siteName`, `board`, `pinMap`, `expectedIntervalSeconds`, `lastSeenAt`).

Mutations:

- `updateMe(input)` — requires `currentPassword`; optional `name`, `newPassword`; `email` is the desired address (enforces uniqueness). After a successful save, the web app signs the user out so they sign in again with a fresh JWT.
- `createAdminUser` / `updateAdminUser` — **`admin` only**; credentials password min 8 chars; `assignedSiteIds` replaces `user_sites` when provided on update; cannot remove the last administrator.
- `resetAdminUserPassword(id, newPassword)` — **`admin` only**; sets `password_hash` without the current password (same min length as create). The **Admin → Edit user** form has a dedicated reset section that calls this mutation.
- `createAdminSite` / `updateAdminSite` — **`admin` only**; optional GPS (`latitude` / `longitude` together), `sensorReporting` as `{ key, enabled }` for **every** catalog sensor; optional `sensorThresholds` as `{ key, minValue, maxValue }` (nullable = use catalog default for that bound); disabling a sensor resolves active anomaly alerts for that sensor’s types
- `createSensorCatalogEntry` / `updateSensorCatalogEntry` / `deleteSensorCatalogEntry` — **`admin` only**; CRUD on `sensor_catalog`. Creating a sensor adds enabled-by-default `site_sensor_catalog` rows for all existing sites. Deleting resolves active MVP anomaly alerts for that key (when applicable) across sites, then cascades join rows.
- `createAdminDevice(input)` — **`admin` only**; generates a fresh API key, stores its SHA-256 hash, returns `{ device, apiKey }` (the plaintext `apiKey` is shown **once**; never persisted anywhere on the server).
- `updateAdminDevice(input)` — **`admin` only**; updates `deviceId`, `name`, `board`, `expectedIntervalSeconds`, and the `pinMap` jsonb (the install wizard writes the chosen sensor→GPIO mapping back here).
- `rotateAdminDeviceApiKey(id)` — **`admin` only**; regenerates the key and returns the new plaintext value (shown once).
- `deleteAdminDevice(id)` — **`admin` only**; removes the device row. Past measurements remain (they nullable-reference `device_id`).
- The `JSON` scalar is used by `AdminDevice.pinMap` and `UpdateAdminDeviceInput.pinMap`; resolvers serialize/deserialize plain JS objects.

GraphQL is intentionally flat. Avoid adding nested abstractions until the UI actually needs them.

## Device ingestion contract

Human- and firmware-oriented guide (ESP32/ESP8266 bring-up, headers, JSON, errors): `docs/esp-device-ingest.md`.

The fastest provisioning path for operators is the **admin device wizard** at `/admin/devices/[id]/install`: it auto-fills the API URL/deviceId/key, lets the user pick which catalog sensors are wired and which GPIO each uses, then patches a JSON config into a reserved 2 KiB region inside the precompiled `firmware.bin` and flashes via `esp-web-tools` over WebSerial. The `seed` script's API key printout remains a developer shortcut.

Endpoint:

```text
POST /ingest
```

Authentication:

- Header: `x-api-key`
- Device API keys are stored as SHA-256 hashes in `devices.api_key_hash`.

Payload:

```json
{
  "deviceId": "device-123",
  "timestamp": "2026-05-01T15:00:00.000Z",
  "readings": {
    "temperature": 24.3,
    "ph": 6.8,
    "waterLevel": 72,
    "dissolvedOxygen": 6.1
  }
}
```

Partial readings are allowed; at least one reading must be present. Each key must exist in `sensor_catalog` (unknown keys are rejected with 400).

## RBAC model

Roles:

- `admin` - full access to every site; no `user_sites` rows required.
- `site_manager` - read/write access to all assigned sites.
- `site_viewer` - read-only access to all assigned sites.

Assignments:

- `user_sites` is a many-to-many join table.
- Both `site_manager` and `site_viewer` users may be assigned to multiple sites.
- `getSites` returns all sites assigned to non-admin users.
- Backend resolvers must call `requireSiteAccess` or otherwise enforce equivalent access checks.
